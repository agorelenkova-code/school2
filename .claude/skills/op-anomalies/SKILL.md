# Skill: Анализ аномалий ОП

## Запуск

```
/op-anomalies [ссылка на Google Таблицу]
```

Skill анализирует выгрузку amoCRM, ищет аномалии в работе отдела продаж и создаёт аналитические листы в той же Google Таблице, а также текстовый отчёт в проекте.

---

## Шаг 0. Подготовка: читай данные перед расчётами

Перед любыми расчётами выполни следующее:

1. Прочитай оба исходных листа **ОП** и **Куратор** целиком.
2. Выпиши фактические названия всех колонок.
3. Сопоставь фактические названия с ожидаемыми (раздел «Обязательные поля»).
4. Если колонка называется иначе, но подходит по смыслу — используй её и отметь в отчёте.
5. Если колонка не найдена — пропусти соответствующий блок анализа и выведи предупреждение.
6. Нормализуй данные: убери лишние пробелы в ФИО менеджеров и названиях причин, нормализуй форматы дат.
7. Зафиксируй дату запуска анализа — она понадобится для расчёта возраста открытых сделок.

Исходные листы **ОП** и **Куратор** не изменять.

## Шаг 1. Основная логика анализа

### Когорта

В базу анализа попадают сделки, где:
- поле «Целевой лид» = «Да»;
- поле «Дата целевого лида» попадает в анализируемый месяц.

Период анализа определяется автоматически по полю «Дата целевого лида».

Если в выгрузке сделки за разные месяцы — не останавливай анализ, но выведи предупреждение:
> «В выгрузке найдены сделки за несколько месяцев, результаты могут быть смешанными.»

Одна строка = одна сделка. Ключ связи между листами: **ID сделки**.

### Обязательные поля

**Лист ОП:**
- ID сделки
- Целевой лид
- Дата целевого лида
- Ответственный
- Дата закрытия
- Причина отсутствия продажи
- Тариф
- БПН
- Текстовые колонки с комментариями (находятся в конце выгрузки)

**Лист Куратор:**
- ID сделки
- Дата продажи
- Продавший менеджер
- Ответственный

Если поле не найдено и похожее по смыслу найти невозможно — пропусти блок анализа, который зависит от этого поля, и обязательно укажи в разделе «Ограничения анализа» отчёта, какие поля не найдены и какие выводы могут быть неполными.

**Если текстовые колонки с комментариями отсутствуют или полностью пусты** — пропусти блок «Спорные НЦ» целиком и выведи предупреждение.

### Определение менеджера

- Продажа: менеджер — «Продавший менеджер» из листа Куратор. Поле «Ответственный» в проданных сделках не использовать как менеджера ОП (там может быть куратор).
- Закрытая неуспешная сделка: менеджер — «Ответственный» из листа ОП.
- Открытая сделка: менеджер — «Ответственный» из листа ОП.

### Цикл сделки

- Старт: «Дата целевого лида».
- Финиш для продажи: «Дата продажи».
- Финиш для закрытой неуспешной: «Дата закрытия».
- Для открытых сделок: дата запуска анализа. В отчёте явно подписывать: «текущий возраст открытой сделки».

Не устанавливай жёсткий порог нормального цикла. Определяй аномалии сравнительно:
- через медиану команды;
- через среднее по команде;
- через выбросы (±1,5 IQR или ±2 стандартных отклонения);
- отдельно по менеджерам, тарифам и отдельным сделкам.

Тот же принцип применяй к конверсии и доле НЦ — не устанавливай жёстких порогов, определяй аномалии относительно команды.

---

## Шаг 2. Конверсии

### Общая конверсия

```
Общая конверсия = Количество продаж / Все целевые лиды
```

Все целевые лиды = сделки с «Целевой лид» = «Да» и «Дата целевого лида» в анализируемом месяце. Открытые сделки входят в знаменатель.

### Конверсия по логике ОП

```
Конверсия ОП = Количество продаж / (Продажи + Закрытые с целевыми причинами отказа)
```

Целевые причины отказа: «Решение (Ц)», «Сроки (Ц)», «Продукт (Ц)».
Нецелевые причины (НЦ): «Контакт (НЦ)», «Невозможно зачислить (НЦ)».

НЦ-причины не входят в знаменатель конверсии ОП.

### Разрыв конверсии

```
Разрыв = Конверсия ОП − Общая конверсия
```

Высокий разрыв у менеджера — сигнал риска: часть целевых лидов могла быть выведена в НЦ-причины для искусственного улучшения конверсии ОП.

Интерпретируй разрыв сравнительно с командой. Менеджеров, у которых разрыв заметно выше медианы команды, выноси в рисковые аномалии.

---

## Шаг 3. Причины отсутствия продажи

Поле: «Причина отсутствия продажи».

Искать:
- менеджеров с высокой долей «Контакт (НЦ)»;
- менеджеров с высокой долей «Невозможно зачислить (НЦ)»;
- менеджеров, у которых много целевых лидов закрыты в НЦ;
- связку «долгий цикл сделки → закрытие в НЦ»;
- связку «высокий разрыв конверсии → высокая доля НЦ»;
- календарные всплески закрытий в НЦ.

---

## Шаг 4. Спорные НЦ по комментариям

Если текстовые колонки с комментариями есть и непусты — анализируй сделки, где причина отказа «Контакт (НЦ)» или «Невозможно зачислить (НЦ)».

Признаки спорного НЦ в тексте комментариев:
- дорого / цена / стоимость
- подумают / подумаю / посоветуются / обсудят
- позже / потом / вернутся / перезвонят
- выбрали другую школу / другой вариант
- не подходит формат / не готовы сейчас
- советуются / обсуждают
- была договоренность о следующем шаге
- обсуждали продукт / тариф / цену / сроки / БПН
- клиент отвечал / было несколько взаимодействий
- причина выглядит как решение клиента, сроки или продукт, а не отсутствие контакта

Если отдельного поля по количеству касаний нет — определяй несколько взаимодействий только по тексту комментариев.

Создай лист **Спорные_НЦ_YYYY-MM** (если существует — **_v2**, **_v3** и т.д.).

Колонки листа:
| Колонка | Описание |
|---|---|
| ID сделки | |
| Менеджер | |
| Текущая причина отсутствия продажи | |
| Тип риска | |
| Пояснение | почему сделка выглядит спорной |
| Фрагмент комментария | признак риска |
| Рекомендация | «Проверить вручную» |

---

## Шаг 5. Тарифы

Поле: «Тариф». Ожидаемые значения: СЗ, БЗ, ВЭБ, ОА, доступ / без учителя.

По каждому тарифу считать:
- количество целевых лидов и продаж;
- общую конверсию и конверсию ОП;
- разрыв конверсий;
- средний и медианный цикл сделки;
- долю отказов, долю НЦ, долю «Контакт (НЦ)» и «Невозможно зачислить (НЦ)»;
- использование БПН;
- незавершённый хвост.

Искать:
- тарифы с циклом выше медианы команды;
- тарифы с долей НЦ выше медианы команды;
- тарифы с конверсией ниже медианы команды;
- менеджеров, у которых по конкретному тарифу результат заметно хуже или лучше команды.

---

## Шаг 6. БПН (бесплатная неделя)

Поле: «БПН». Значение «БПН» в поле = БПН назначена.

Дополнительно проверяй текстовые комментарии на упоминания: БПН, бесплатная неделя, пробная неделя, тестовая неделя, бесплатный период, пробный период.

**Логика приоритета при конфликте поля и комментария:**
- Поле «БПН» заполнено, комментарий не упоминает БПН → считать БПН назначенной, пометить в листе БПН как «БПН без пояснения в комментарии» и вынести на ручную проверку.
- Поле «БПН» пустое, комментарий содержит признак БПН → считать как «БПН по комментарию», пометить как «поле не заполнено» и вынести на ручную проверку.
- Поле заполнено и комментарий подтверждает → стандартный учёт.

Искать:
- кто из менеджеров назначает БПН чаще или реже медианы команды;
- как БПН связан с конверсией, циклом сделки и отказами;
- связку «БПН → долгий цикл → отказ»;
- связку «БПН → Контакт (НЦ)».

---

## Шаг 7. Незавершённый хвост

Хвост — сделки, которые:
- попали в когорту по «Дата целевого лида»;
- «Целевой лид» = «Да»;
- нет «Дата продажи»;
- нет «Дата закрытия».

Анализировать:
- количество таких сделок;
- у каких менеджеров;
- текущий возраст открытых сделок (подписывать явно);
- сделки-выбросы по длительности;
- по каким тарифам чаще зависают;
- наличие БПН в хвосте;
- менеджеры с хвостом заметно выше медианы команды с учётом базы.

Количество лидов по менеджеру использовать только как контекст — не как самостоятельный вывод о нагрузке.

---

## Шаг 8. Календарные аномалии

Даты: «Дата закрытия» для неуспешных, «Дата продажи» для продаж.

Анализировать:
- всплески закрытий по дням;
- всплески НЦ, «Контакт (НЦ)», «Невозможно зачислить (НЦ)»;
- рост закрытий в конце месяца;
- дни с долей НЦ выше обычной;
- дни с резким ростом закрытий без продажи.

Сравнивать по неделям месяца: 1-я, 2-я, 3-я, последняя. Конец месяца выделять отдельно, если есть всплеск.

Анализировать и отдельных менеджеров, и команду целиком.

---

## Шаг 9. Рисковые аномалии

Искать **все** рисковые аномалии. Группировать по критичности:

**Критично:**
- высокий разрыв конверсии;
- долгий цикл + закрытие в «Контакт (НЦ)»;
- долгий цикл + закрытие в «Невозможно зачислить (НЦ)»;
- массовые закрытия в НЦ в отдельные дни;
- спорные НЦ по комментариям;
- высокая доля НЦ при достаточной базе.

**Средне:**
- менеджер заметно отличается от команды по структуре отказов;
- аномально частое или редкое использование БПН;
- повышенный незавершённый хвост;
- выбросы по тарифам;
- необычная структура закрытий по неделям.

**Низко:**
- единичные отклонения;
- отклонения на маленькой базе;
- случаи, требующие ручной проверки без явного системного риска.

**Правило малой базы:** если у менеджера менее 5 целевых лидов — все выводы по нему помечать: «низкая база, требуется ручная проверка». Не делать системных выводов.

## Шаг 10. Положительные аномалии

Искать лучшие практики по шести типам:

1. **Низкий разрыв конверсий** — менеджер меньше выводит целевые лиды в НЦ.
2. **Короткий цикл сделки** — средний или медианный цикл ниже команды без роста НЦ.
3. **Низкая доля «Контакт (НЦ)»** — реже команды закрывает целевые лиды в «Контакт (НЦ)».
4. **Высокая общая конверсия без перекоса в НЦ** — результат выглядит устойчивым.
5. **Эффективное использование БПН** — сделки с БПН дают более высокую конверсию или меньший цикл, чем в среднем по команде.
6. **Низкий незавершённый хвост** — меньше зависших сделок при сопоставимой базе.

Создай лист **Положительные_аномалии_YYYY-MM** (если существует — **_v2**, **_v3** и т.д.).

Колонки листа:
| Колонка | Описание |
|---|---|
| Менеджер | |
| Тип положительной аномалии | |
| Показатель менеджера | |
| Показатель команды | |
| Отличие от команды | |
| Возможная сильная практика | |
| Рекомендация | «Изучить / масштабировать» |

## Шаг 12. Главный управленческий отчёт

Лист «Отчет_YYYY-MM» — главный. Показывать только управленчески значимые выводы, не операционную сводку.

**Структура:**

### 1. Общий итог по месяцу
- количество целевых лидов, продаж, закрытых без продажи, открытых в хвосте;
- общая конверсия, конверсия ОП, разрыв конверсий.

### 2. Рисковые зоны
- ключевые аномалии по критичности;
- менеджеры и тарифы с рисками;
- всплески закрытий;
- спорные НЦ;
- БПН-риски;
- хвост.

### 3. Положительные аномалии
- менеджеры с лучшими практиками по каждому типу.

### 4. Сделки на ручную проверку
Показывать ID сделок. Особенно:
- долгий цикл + НЦ;
- спорный «Контакт (НЦ)»;
- спорное «Невозможно зачислить (НЦ)»;
- комментарии противоречат причине отказа;
- БПН в комментарии, но поле не заполнено.

### 5. Рекомендации
- что проверить вручную;
- по каким менеджерам провести разбор;
- где нужна калибровка Ц / НЦ;
- где риск искусственного завышения конверсии;
- где проверить корректность БПН;
- какие лучшие практики стоит масштабировать.

## Стиль выводов

Пиши управленческим языком. Не просто перечисляй цифры — объясняй, что это значит для руководителя ОП.

Формат каждого вывода:
- **Факт** — что обнаружено и в каких числах.
- **Почему важно** — какой риск или возможность это несёт.
- **Гипотеза** — что могло привести к этому.
- **Что проверить** — конкретные ID сделок или действия.
- **Рекомендация** — управленческое действие.

**Пример:**

> **Факт:** У менеджера Иванов разрыв между конверсией ОП (38%) и общей конверсией (21%) составляет 17 п.п. — выше медианы команды (7 п.п.) более чем вдвое.
>
> **Почему важно:** Высокий разрыв указывает на возможный вывод целевых лидов из базы расчёта через НЦ-причины.
>
> **Гипотеза:** Причина «Контакт (НЦ)» могла использоваться вместо целевых причин отказа.
>
> **Что проверить:** Сделки ID 123, 456, 789.
>
> **Рекомендация:** Провести калибровку по правилам выбора причины отсутствия продажи.

## Технические требования к расчётам

- Все проценты выводить в формате: **12 сделок / 18,5%**
- Пустая «Дата продажи» ≠ продажа. Пустая «Дата закрытия» ≠ закрытие.
- Нормализовать: даты, ФИО менеджеров, названия причин отказа, пробелы.
- Учитывать пустые значения во всех расчётах.
